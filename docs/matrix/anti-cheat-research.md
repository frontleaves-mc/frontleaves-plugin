## 1. 调研概述

### 1.1 调研目的

Matrix 子系统采用**外部进程检测架构**，Go 服务独立于 MC 服务端运行。本调研旨在回答三个问题：

1. 外部进程架构在 Minecraft 反作弊生态中有无先例？成熟度如何？
2. 主流开源项目的检测方法有哪些可以直接借鉴？
3. 本项目的定位和未来方向应该是什么？

### 1.2 调研范围

- **开源项目**：GrimAC、NoCheatPlus、AsyncAnticheat、AntiCheatAddition、LightAntiCheat
- **闭源参考**：Polar、Vulcan、Themis
- **学术文献**：LSTM Aimbot Detection、Checkmate 研究

### 1.3 调研方法

- GitHub 仓库代码审查与 README 分析
- SpigotMC / PaperMC 社区讨论与用户反馈
- 学术论文检索（IEEE、ACM）
- 各项目 Wiki 文档与配置说明

### 2.2 各项目简评

#### GrimAC — 当前开源标杆

GrimAC 是目前最活跃的开源反作弊项目。它的核心创新是**预测引擎（Prediction Engine）**：服务端模拟一个完整的客户端移动模型，将玩家上报的位置与预测位置逐一比对。如果偏差超过容忍阈值，判定为作弊。

- 优势：检测精度高，误报率低；支持 1.8-1.21 多版本协议
- 劣势：预测引擎计算开销较大，在低 TPS 服务器上可能出现检测漂移
- 社区规模大，Issue 响应快

#### NoCheatPlus — 老牌经典

NCP 是 Minecraft 反作弊领域的元老级项目，采用传统的阈值检测方式。通过设置各项参数的上下限，判断玩家行为是否异常。内置战斗、移动、方块等多类检测模块。

- 优势：成熟稳定，配置灵活，文档完善
- 劣势：检测模型较为陈旧，对新版客户端特性支持滞后；在 Paper 1.20+ 上存在兼容性问题
- 维护节奏放缓，社区逐渐向 GrimAC 迁移

#### AsyncAnticheat — 外部架构先例

这个项目非常值得关注。它用 Rust 编写，将检测逻辑完全放在外部进程中，通过自定义协议与 MC 服务端通信。这正是本项目（Matrix）采用的架构模式。

- 优势：零 TPS 影响，检测逻辑与游戏主线程完全解耦；Rust 带来的内存安全和性能
- 劣势：项目活跃度极低，处于实验阶段；社区信任度不足
- 对本项目的意义：证明外部进程架构在技术上可行，但需要更完善的工程实践来建立信任

#### Polar — 闭源标杆

Polar 是目前闭源反作弊中口碑较好的产品。它采用混合异步架构，部分检测在 Netty 线程完成，部分在 Bukkit 线程。包级别的分析能力使其在高版本上保持了较好的检测覆盖。

- 特色：延迟补偿机制（Latency Compensation），考虑网络延迟对检测的影响
- 定价：按服务器规模订阅

#### Vulcan — 高精度检测

Vulcan 以多层检测和启发式算法著称。它在传统阈值检测的基础上加入了行为模式分析，对 KillAura 和 Speed 的检测准确度较高。

- 特色：多维度交叉验证（位置 + 朝向 + 包时序）
- 劣势：纯内部插件架构，对 TPS 有直接压力

## 3. 架构模式对比

反作弊的架构模式直接决定了性能影响、检测精度和部署复杂度。

### 3.1 三种模式

```
模式 A：纯内部插件检测
┌──────────────────────────────┐
│        MC 服务端 (JVM)        │
│  ┌──────┐  ┌──────────────┐  │
│  │ 游戏主线程 │  │ 反作弊检测线程 │  │
│  └──────┘  └──────────────┘  │
│         共享 JVM 资源          │
└──────────────────────────────┘
代表：NoCheatPlus、Vulcan、AAC

模式 B：混合异步检测
┌──────────────────────────────┐
│        MC 服务端 (JVM)        │
│  ┌──────┐  ┌──────────────┐  │
│  │ 游戏主线程 │  │ Netty 异步线程 │  │
│  └──────┘  └──────┬───────┘  │
└────────────────────┼─────────┘
                     │ 部分卸载到独立线程
代表：GrimAC、Polar、Themis

模式 C：外部进程检测
┌──────────────────┐    ┌──────────────────┐
│  MC 服务端 (JVM)  │    │   外部进程 (Go)   │
│  ┌──────────────┐ │    │  ┌──────────────┐ │
│  │ 仅采集上报    │ │───→│  │ 完整检测逻辑  │ │
│  └──────────────┘ │gRPC│  └──────────────┘ │
└──────────────────┘    └──────────────────┘
代表：AsyncAnticheat、本项目 (Matrix)
```

### 3.2 模式对比表

| 维度 | 模式 A (纯内部) | 模式 B (混合异步) | 模式 C (外部进程) |
|---|---|---|---|
| **TPS 影响** | 高 | 中低 | 零 |
| **检测延迟** | 最低 (同进程) | 低 (线程间通信) | 较高 (网络传输) |
| **部署复杂度** | 低 (拖入 plugins) | 低 (拖入 plugins) | 高 (需要额外部署) |
| **计算资源** | 共享 JVM 堆 | 部分独立线程 | 完全独立进程 |
| **数据访问** | 直接调用 Bukkit API | 直接 + 异步混合 | 仅能获取上报数据 |
| **检测精度** | 依赖 TPS 稳定性 | 较好 | 依赖数据完整性 |
| **横向扩展** | 不可扩展 | 有限扩展 | 可独立扩容 |
| **协议独立性** | 绑定 MC 版本 | 部分独立 | 完全独立 |

### 3.3 模式 C 的核心挑战

本项目选择模式 C，需要正视以下挑战：

1. **数据完整性**：外部进程只能处理 MC 端选择上报的数据，无法像内部插件那样随时查询任意 Bukkit API。Proto 契约的覆盖范围直接决定检测能力的上限。
2. **检测延迟**：gRPC Stream 传输 + Redis 缓冲 + 协程处理，整体延迟在 50-200ms 范围内。对于 Speed/Fly 等持续性作弊可以容忍，但对 Aimbot/KillAura 等需要瞬时响应的检测有局限。
3. **游戏状态同步**：MC 端的方块更新、实体生成/消失等状态变化需要通过事件流同步，信息丢失可能导致误判。

## 5. TPS 处理策略对比

TPS（Ticks Per Second）是 MC 服务端的核心性能指标。标准 TPS 为 20，低于 15 时玩家体验明显下降。反作弊的检测逻辑如果与游戏主线程耦合，TPS 下降会导致检测失准甚至误判。

### 5.1 各项目的 TPS 应对策略

| 项目 | TPS 依赖 | 应对策略 | TPS 下降时的表现 |
|---|---|---|---|
| **GrimAC** | 中等 | 基于游戏 Tick 的预测引擎，自动适配 TPS 变化 | TPS 极低时预测精度下降，可能增加误报 |
| **NCP** | 高 | Tasklag 补偿机制，检测间隔按 TPS 动态调整 | TPS < 15 时部分检测自动放宽阈值 |
| **Vulcan** | 中高 | 内置 TPS 监测，动态调整检测灵敏度 | TPS 波动时进入宽松模式 |
| **Polar** | 低 | 延迟补偿 + 独立计时器 | TPS 影响较小 |
| **AsyncAnticheat** | 无 | 外部进程，独立时间基准 | 不受 TPS 影响 |
| **Matrix (本项目)** | 无 | 外部进程，使用系统时钟独立计时 | 不受 TPS 影响 |

### 5.2 关键洞察

**传统反作弊的根本矛盾**：当服务器 TPS 下降时，反作弊需要同时应对两个问题：

1. **检测精度下降**：基于 Tick 的计算假设每 Tick 固定 50ms，但 TPS 下降时实际间隔变大
2. **自身的性能开销加剧 TPS 下降**：检测逻辑越复杂，对主线程的压力越大，形成恶性循环

**模式 C 的天然优势**：外部进程完全脱离 JVM，使用操作系统的系统时钟而非游戏 Tick。Matrix 的 500ms manage 循环以 `time.Ticker` 驱动，不受 MC TPS 影响。这消除了上述矛盾。

## 6. 学术研究参考

### 6.1 LSTM Aimbot Detection

一篇值得关注的研究使用长短期记忆网络（LSTM）检测 FPS 游戏中的 Aimbot 作弊行为。

- **方法**：采集玩家的鼠标移动轨迹数据（朝向变化序列），训练 LSTM 模型区分人类操作和自动化脚本
- **结果**：在测试集上达到 98.5% 的检测准确率
- **启示**：Aimbot 检测不一定依赖包分析，行为模式分析同样有效。Matrix 的 TelemetryTick 提供 yaw/pitch 时序数据，可以作为行为分析的特征输入
- **局限**：训练数据来自 FPS 游戏，MC 的战斗节奏不同，需要重新收集训练数据

### 6.2 Checkmate 研究

Checkmate 是一个面向 Minecraft 的反作弊研究框架，探索了多种机器学习方法在作弊检测中的应用。

- **覆盖范围**：Speed、Fly、KillAura 等主要作弊类型
- **方法**：对比了 SVM、随机森林、神经网络等多种模型
- **结论**：对于持续性作弊（Speed、Fly），传统阈值检测已足够；对于伪装性强的作弊（KillAura、Aimbot），机器学习方法显著优于规则引擎
- **启示**：Matrix 的架构天然适合做行为数据的收集和模型训练。Proto 契约已经覆盖了足够的特征维度

### 6.3 其他方向

- **统计分析方法**：对玩家行为数据做正态分布检验，偏移超过 N 个标准差即为异常。适合 X-Ray 检测（矿石发现率的异常分布）
- **时序异常检测**：使用 Isolation Forest 或 AutoEncoder 对 TelemetryTick 时序数据做异常检测，无需标注数据

## 7. 对本项目的启示

### 7.1 外部架构的核心优势

Matrix 选择模式 C（外部进程检测）带来的优势是结构性的，不是调参能弥补的：

1. **零 TPS 影响**：检测计算在独立进程中完成，MC 服务端的 Tick 循环完全不承担反作弊开销。100 个玩家的检测不会让 TPS 降低哪怕 0.1
2. **独立计算资源**：Go 进程可以独占 CPU 核心，不受 GC 停顿和 Java 线程调度的影响。检测延迟可预测、可控制
3. **横向扩展能力**：当玩家规模增长时，Go 端可以独立扩容（多实例、更强的 CPU），不需要修改 MC 服务端配置
4. **协议独立性**：检测逻辑不绑定具体的 MC 版本。只要 Proto 契约稳定，MC 端升级不影响 Go 端

### 7.2 可借鉴的检测方法

从 GrimAC 和 Polar 的实践中，以下方法值得引入：

| 方法 | 来源 | 适用场景 | 引入难度 |
|---|---|---|---|
| **预测引擎** | GrimAC | Movement 检测（Speed/Fly/NoFall） | 高，需要完整的 MC 物理模拟 |
| **延迟补偿** | Polar | 所有涉及位置的检测 | 中，需要精确的 RTT 测量 |
| **多维度交叉验证** | Vulcan | Combat 检测（KillAura/Reach） | 中，已有位置+朝向数据 |
| **统计分布检验** | 通用 | X-Ray 检测 | 低，基于 BlockBreakEvent 聚合 |
| **时序模式分析** | 学术 | AutoClicker/Aimbot 检测 | 中，需要积累训练数据 |

### 7.3 推荐的实现优先级

基于价值和实现成本的平衡：

1. **Phase 1 (已完成)**：Speed 检测，基于 TelemetryTick 位置变化 + 可配置阈值
2. **Phase 2 (推荐下一步)**：Reach 检测，基于 EntityDamageEvent 攻击者-目标距离计算。Proto 已定义完整数据，实现成本低，价值高
3. **Phase 3**：X-Ray 检测，基于 BlockBreakEvent 的矿石发现率统计分布。纯统计方法，不需要模拟物理
4. **Phase 4**：AutoClicker 检测，基于 EntityDamageEvent 的点击间隔统计分析
5. **Phase 5 (远期)**：引入机器学习模型做 KillAura/Aimbot 检测。需要先积累足够的标注数据

### 7.4 与内部插件的协作边界

Matrix 不应该取代所有内部检测。合理的分工是：

- **Matrix (Go 端)**：统计聚合、行为模式分析、跨会话追踪、历史数据比对。擅长需要大量计算或历史数据的检测
- **MC 插件端**：需要瞬时响应的检测（如 Scaffold 的方块放置合法性验证）、需要直接访问 Bukkit API 的检测（如 IllegalEnchant 的物品属性校验）

这种分工让两端各取所长，而不是互相替代。
